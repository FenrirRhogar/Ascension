Technical Architecture and Research for a Unity 2026 Split-Screen Cooperative Dungeon CrawlerExecutive Summary and Game Design DocumentThe development of a robust, local split-screen multiplayer application within the Unity 2026 engine demands a highly modular, scalable, and performant architectural foundation. To satisfy the rigorous technical constraints outlined by the Graphics Unity Project 2026 , the software architecture must seamlessly integrate the Unity New Input System for simultaneous local device polling, utilize ScriptableObject data containers for memory-efficient state management, and employ procedural generation algorithms to synthesize dynamic environments. The integration of these systems ensures that the application not only meets the fundamental requirements of a shared-screen cooperative experience but also scales efficiently without incurring technical debt.The following Game Design Document (GDD) outlines the core parameters, thematic setting, and entity classifications required to successfully implement the "Fantasy Dungeon" theme while fulfilling all academic project criteria, including basic requirements, extra functionalities, and potential bonus implementations.Thematic Setting and Core ObjectivesThe application, internally designated as Echoes of the Obsidian Citadel, functions as a cooperative (Co-op) 3D dungeon crawler where two users must navigate procedurally generated subterranean labyrinths. Set in a dark fantasy epoch, players assume the roles of specialized adventurers attempting to breach the deeper sanctums of a ruined fortress. The primary objective requires players to collaboratively survive hostile encounters, gather necessary consumables, circumvent physical obstacles, and defeat the sanctum's artificial intelligence (AI) guardians to unlock the exit to the subsequent procedural tier.The game loop is inherently reliant on interdependent mechanics. Neither player possesses the complete toolkit required to survive the procedural generation alone; they must rely on the asymmetrical class design to overcome obstacles.Entity Classification and MechanicsThe application of a modular design philosophy necessitates the strict categorization of in-game entities. If players collide with Level Exit $\rightarrow$ Increment Level Index $\rightarrow$ Return to State 4.Modular Architecture: Scriptable Objects and Class ManagementA primary technical challenge in developing an application featuring four highly distinct playable classes is mitigating code duplication and avoiding monolithic, heavily branched conditional logic. Traditional approaches involving deeply nested switch statements or massive inheritance trees attached to MonoBehaviour scripts often lead to severe memory fragmentation and unmaintainable codebases.To achieve a highly modular and performant architecture, the system relies on the ScriptableObject pattern. ScriptableObjects are serializable data containers that exist at the project asset level, functioning entirely independently of scene instantiations. By utilizing these objects, developers can isolate gameplay data—such as maximum health, locomotion speed multipliers, and attack damage integers—from behavioral execution logic. Furthermore, applying inheritance directly to ScriptableObjects permits the creation of a plug-and-play ability system where a unified PlayerController and CombatSystem can govern all four classes simultaneously.Base Class Data StructuresThe foundation of the modular class system is an abstract data container that defines the universal parameters shared by the Wizard, Assassin, Fighter, and Cleric. This base class forces derived objects to implement their own specific combat execution methods, guaranteeing polymorphism.C#using UnityEngine;

// Defines the base template for any playable class in the game
public abstract class CharacterClassSO : ScriptableObject
{
   
    public string className;
    public int maxHealth;
    public float baseMovementSpeed;
    public float attackCooldown;

    [Header("Visual Assets")]
    public GameObject characterModelPrefab;
    public Sprite classIcon;

    public abstract void ExecuteAction(Transform playerTransform, Animator animator);
}
By defining ExecuteAction as an abstract method, the architecture mandates that specialized assets provide their own unique implementation. The data and the logic to execute that data are encapsulated entirely within the asset file. The following boilerplate code demonstrates how the Wizard and Fighter classes diverge in their mechanical execution.C#using UnityEngine;


public class WizardClassSO : CharacterClassSO
{
   
    public GameObject magicProjectilePrefab;
    public float projectileSpeed;

    public override void ExecuteAction(Transform playerTransform, Animator animator)
    {
        animator.SetTrigger("CastSpell");
        
        // Instantiate a physics-based projectile for ranged AOE damage
        Vector3 spawnPos = playerTransform.position + (playerTransform.forward * 1.5f) + Vector3.up;
        GameObject projectile = Instantiate(magicProjectilePrefab, spawnPos, playerTransform.rotation);
        
        Rigidbody rb = projectile.GetComponent<Rigidbody>();
        if(rb!= null)
        {
            rb.AddForce(playerTransform.forward * projectileSpeed, ForceMode.Impulse);
        }
    }
}

[CreateAssetMenu(fileName = "NewFighterClass", menuName = "Classes/Fighter")]
public class FighterClassSO : CharacterClassSO
{
   
    public float meleeRange;
    public int meleeDamage;
    public LayerMask enemyLayer;

    public override void ExecuteAction(Transform playerTransform, Animator animator)
    {
        animator.SetTrigger("MeleeSwing");
        
        // Implementation of Physics.OverlapSphere to detect enemies in melee range
        Vector3 strikeCenter = playerTransform.position + (playerTransform.forward * 1.0f);
        Collider hitEnemies = Physics.OverlapSphere(strikeCenter, meleeRange, enemyLayer);
        
        foreach (Collider enemy in hitEnemies)
        {
            EnemyAI aiComponent = enemy.GetComponent<EnemyAI>();
            if (aiComponent!= null)
            {
                aiComponent.TakeDamage(meleeDamage);
            }
        }
    }
}
The Unified Player Controller and Combat SystemWith the data and unique combat logic encapsulated within the asset files, the components attached to the player's root GameObject remain incredibly lightweight. To fulfill the specific architectural mandate of separating movement logic from offensive logic, the design utilizes two distinct MonoBehaviour scripts: the PlayerController and the CombatSystem.The PlayerController acts exclusively as a physical vessel. It parses spatial inputs and translates the CharacterController component based on the speed variables dictated by the assigned ScriptableObject.C#using UnityEngine;
using UnityEngine.InputSystem;


public class PlayerController : MonoBehaviour
{
   
    public CharacterClassSO currentClass;
    
    private CharacterController charController;
    public Animator animator { get; private set; }
    
    private Vector2 movementInput;
    public int currentHealth { get; set; }

    void Awake()
    {
        charController = GetComponent<CharacterController>();
    }

    public void InitializeClass(CharacterClassSO selectedClass)
    {
        currentClass = selectedClass;
        currentHealth = currentClass.maxHealth;
        
        // Instantiate the specific 3D model assigned to this class
        GameObject model = Instantiate(currentClass.characterModelPrefab, transform);
        animator = model.GetComponent<Animator>();
    }

    public void OnMove(InputValue value)
    {
        movementInput = value.Get<Vector2>();
    }

    void Update()
    {
        if (currentClass == null) return;

        Vector3 move = new Vector3(movementInput.x, 0, movementInput.y);
        
        // Apply gravity and locomotion translation
        charController.Move((move * currentClass.baseMovementSpeed + Physics.gravity) * Time.deltaTime);
        
        if (move!= Vector3.zero)
        {
            gameObject.transform.forward = move;
            animator.SetBool("IsWalking", true);
        }
        else
        {
            animator.SetBool("IsWalking", false);
        }
    }
}
Conversely, the CombatSystem subscribes to the action inputs generated by the Unity New Input System. When a player triggers the attack command, the CombatSystem verifies cooldown thresholds and then blindly delegates the execution back to the ScriptableObject.C#using UnityEngine;
using UnityEngine.InputSystem;


public class CombatSystem : MonoBehaviour
{
    private PlayerController playerController;
    private float lastAttackTime;

    void Awake()
    {
        playerController = GetComponent<PlayerController>();
    }

    public void OnAttack(InputValue value)
    {
        if (playerController.currentClass == null) return;

        if (value.isPressed && Time.time >= lastAttackTime + playerController.currentClass.attackCooldown)
        {
            // The combat system does not know what the attack is; it delegates to the SO
            playerController.currentClass.ExecuteAction(this.transform, playerController.animator);
            lastAttackTime = Time.time;
        }
    }
}

public class MultiplayerGameManager : MonoBehaviour
{
    public PlayerInputManager inputManager;
    public Transform spawnPoints;
    
    // Arrays representing the pre-selected classes from the menu
    public CharacterClassSO availableClassesP1;
    public CharacterClassSO availableClassesP2;
    
    private int playersJoined = 0;

    void OnEnable()
    {
        inputManager.onPlayerJoined += HandlePlayerJoin;
    }

    void OnDisable()
    {
        inputManager.onPlayerJoined -= HandlePlayerJoin;
    }

    private void HandlePlayerJoin(PlayerInput playerInput)
    {
        if (playersJoined >= 2) return;

        // Route the newly spawned player to the correct physical location
        playerInput.transform.position = spawnPoints[playersJoined].position;
        
        PlayerController pController = playerInput.GetComponent<PlayerController>();

        // Hardware Fallback Logic: Detect if the input stems from a keyboard
        bool isKeyboard = playerInput.devices.Count > 0 && playerInput.devices is Keyboard;

        if (playersJoined == 0)
        {
            // Player 1 Initialization
            pController.InitializeClass(availableClassesP1); // Default to first choice (Wizard)
            
            if (isKeyboard)
            {
                playerInput.SwitchCurrentControlScheme("KeyboardLeft", Keyboard.current);
            }
        }
        else if (playersJoined == 1)
        {
            // Player 2 Initialization
            pController.InitializeClass(availableClassesP2); // Default to first choice (Fighter)
            
            if (isKeyboard)
            {
                // Force the right-side keyboard mapping to prevent P1 interference
                playerInput.SwitchCurrentControlScheme("KeyboardRight", Keyboard.current);
            }
            // If the device is a Gamepad, the Input System natively resolves it via the Gamepad scheme
        }
        
        playersJoined++;
    }
}

            if (isTopDown)
            {
                camTransform.localPosition = topDownCameraOffset;
                camTransform.localRotation = Quaternion.Euler(90, 0, 0);

            else
            {
                camTransform.localPosition = defaultCameraOffset;
                camTransform.localRotation = Quaternion.Euler(30, 0, 0);
            }

using UnityEngine.AI;


public class EnemyAI : MonoBehaviour
{
    public enum State { Idle, Patrol, Chase, Attack }
    public State currentState = State.Idle;

    private NavMeshAgent agent;
    private Transform targetPlayer; 
    private Animator animator;

    [Header("AI Parameters")]
    public float detectionRadius = 10f;
    public float attackRadius = 2f;
    public int attackDamage = 15;
    
    private float stateTimer = 0f;

    void Start()
    {
        agent = GetComponent<NavMeshAgent>();
        animator = GetComponent<Animator>();
    }

    void Update()
    {
        switch (currentState)
        {
            case State.Idle:
                animator.SetBool("IsMoving", false);
                stateTimer += Time.deltaTime;
                if (stateTimer > 2f) 
                {
                    stateTimer = 0f;
                    currentState = State.Patrol;
                }
                CheckForPlayer();
                break;

            case State.Patrol:
                animator.SetBool("IsMoving", true);
                if (!agent.pathPending && agent.remainingDistance < 0.5f)
                {
                    // Generate a random Cartesian coordinate for the patrol route
                    Vector3 randomDir = Random.insideUnitSphere * 5f;
                    randomDir += transform.position;
                    
                    NavMeshHit hit;
                    if(NavMesh.SamplePosition(randomDir, out hit, 5f, 1))
                    {
                        agent.SetDestination(hit.position);
                    }
                }
                CheckForPlayer();
                break;

            case State.Chase:
                if (targetPlayer!= null)
                {
                    animator.SetBool("IsMoving", true);
                    agent.SetDestination(targetPlayer.position);
                    
                    if (Vector3.Distance(transform.position, targetPlayer.position) <= attackRadius)
                    {
                        currentState = State.Attack;
                    }
                    else if (Vector3.Distance(transform.position, targetPlayer.position) > detectionRadius)
                    {
                        targetPlayer = null;
                        currentState = State.Patrol;
                    }
                }
                break;

            case State.Attack:
                agent.SetDestination(transform.position); // Halt agent translation
                animator.SetBool("IsMoving", false);
                transform.LookAt(new Vector3(targetPlayer.position.x, transform.position.y, targetPlayer.position.z));
                
                stateTimer += Time.deltaTime;
                if (stateTimer > 1.5f) // Time between strikes
                {
                    animator.SetTrigger("MeleeSwing");
                    targetPlayer.GetComponent<PlayerController>().currentHealth -= attackDamage;
                    stateTimer = 0f;
                }

                if (Vector3.Distance(transform.position, targetPlayer.position) > attackRadius)
                {
                    currentState = State.Chase;
                }
                break;
        }
    }

    private void CheckForPlayer()
    {
        // Query the physics engine for all potential targets
        Collider hits = Physics.OverlapSphere(transform.position, detectionRadius);
        float closestDistance = Mathf.Infinity;

        foreach (Collider hit in hits)
        {
            if (hit.CompareTag("Player"))
            {
                float dist = Vector3.Distance(transform.position, hit.transform.position);
                if (dist < closestDistance)
                {
                    closestDistance = dist;
                    targetPlayer = hit.transform;
                    currentState = State.Chase;
                }
            }
        }
    }
}
Procedural Map Generation and Spatial SortingTo secure the extra 20% grading requirement for Procedural Map Generation , the environment cannot rely entirely on static, hand-placed prefabs. The algorithm must procedurally scatter physical obstacles, interactive consumables, and hostile NPCs across the playing field. Crucially, it must guarantee mathematically that entities do not spawn overlapping each other or clipping through level boundaries.The Spatial Sampling AlgorithmWhile highly advanced techniques like Poisson Disc Sampling offer mathematically perfect distributions and tight packing , a highly effective and far less computationally expensive heuristic for this specific academic scope involves randomized coordinate generation validated through Physics Application Programming Interfaces (APIs). Utilizing Physics.CheckSphere creates a lightweight placement algorithm that queries the spatial volume before approving an instantiation command.C#using UnityEngine;

public class ProceduralGenerator : MonoBehaviour
{
   
    public GameObject obstaclePrefabs;
    public GameObject npcPrefabs;
    public GameObject consumablePrefabs;
    
    public int numberOfObstacles = 20;
    public int numberOfNPCs = 5;
    public int numberOfConsumables = 10;
    
   
    public Vector2 mapSize = new Vector2(40f, 40f);
    public float overlapCheckRadius = 1.5f;
    public LayerMask collisionMask;

    void Start()
    {
        GenerateObjects(obstaclePrefabs, numberOfObstacles);
        GenerateObjects(npcPrefabs, numberOfNPCs);
        GenerateObjects(consumablePrefabs, numberOfConsumables);
    }

    void GenerateObjects(GameObject prefabArray, int targetCount)
    {
        int objectsSpawned = 0;
        int attempts = 0; // Safety breakout to prevent infinite while-loops

        while (objectsSpawned < targetCount && attempts < 1000)
        {
            attempts++;
            
            // Generate a random Cartesian coordinate within the specified floor bounds
            Vector3 randomPos = new Vector3(
                Random.Range(-mapSize.x / 2, mapSize.x / 2),
                0.5f, // Offset slightly above the floor plane
                Random.Range(-mapSize.y / 2, mapSize.y / 2)
            );

            // Query the physics engine: Is this spherical volume empty?
            if (!Physics.CheckSphere(randomPos, overlapCheckRadius, collisionMask))
            {
                // Select a random prefab from the provided array parameter
                GameObject selectedPrefab = prefabArray;
                
                // Randomize the Y-axis rotation for visual variance
                Quaternion randomRot = Quaternion.Euler(0, Random.Range(0, 360), 0);
                
                Instantiate(selectedPrefab, randomPos, randomRot, this.transform);
                objectsSpawned++;
            }
        }
    }
}
NavMesh Integration for Procedural GeometryA critical complication introduced by procedural generation is its catastrophic impact on pre-baked Artificial Intelligence navigation data. If a massive boulder is spawned randomly in the center of a corridor, the statically baked NavMesh will not recognize the new collision data. Consequently, NPCs will attempt to walk directly through the object, breaking immersion and violating physical constraints.To rectify this without necessitating a massive recalculation of the entire NavMesh at runtime, every generated obstacle prefab must contain a NavMeshObstacle component with the Carve parameter explicitly enabled. When the procedural generator instantiates the object, this component automatically calculates the local geometry and cuts a boolean hole into the existing NavMesh graph.For purely stationary procedural obstacles like trees or fallen pillars, it is a strict technical recommendation to ensure the Carve Only Stationary flag is checked within the inspector. This forces the engine to calculate the pathing void only once upon instantiation, drastically minimizing CPU overhead during active gameplay frames.User Interface (UI) Design and ImplementationManaging the User Interface in a split-screen configuration demands a specialized Canvas approach. Standard Screen Space - Overlay canvases render UI elements globally across the entire monitor, indiscriminately disregarding camera viewports, which would result in health bars stretching across both players' screens.To provide truly independent Heads Up Displays (HUDs) containing individual Healthbars and Inventory counts , the system utilizes the Screen Space - Camera render mode.Within the Player Prefab, a dedicated UI Canvas is instantiated as a child object.The Canvas Render Mode is set to Screen Space - Camera.The Render Camera slot is dynamically assigned via script to the player's specific split-screen camera upon instantiation.This guarantees that Player 1's health bar inherently scales and anchors exclusively to the left side of the screen, and Player 2's UI anchors to the right, mitigating visual overlap errors entirely.Main Menu ArchitectureThe overarching requirement for UI also specifies the inclusion of a robust Main Menu. The architecture for this involves a separate Scene (e.g., MainMenuScene) containing an Overlay Canvas with the following logic bindings tied to Unity UI Buttons:Start Game Button: Triggers SceneManager.LoadScene("GameplayScene") to initiate the procedural game loop and hardware polling.Tutorial Button: Toggles the active state of a UI Panel overlay containing textual directions delineating the WASD and Gamepad input maps.Credits Button: Toggles a UI Panel listing all asset attributions and origins. This is a critical requirement for maintaining academic integrity regarding the use of third-party 3D models and audio files.Quit Button: Executes the Application.Quit() method. For testing purposes, an #if UNITY_EDITOR preprocessor directive utilizing UnityEditor.EditorApplication.isPlaying = false should be included to halt the editor runtime.Asset Sourcing and Thematic CohesionTo achieve the mandated "Fantasy Dungeon" aesthetic required for the classes and NPCs, the project must prioritize assets that offer unified visual styles and functional, retargetable rigging parameters.
